## Overview

**jk-nb-inbox-digest** 是 jk-nb 個人知識庫的自動化整理工具。它專門處理 inbox.md 的升格工作流程：定期掃描待升格條目、判斷是否仍有價值、決定目標 wiki 位置、執行遷移、並在 inbox.md 中標記已處理的條目為刪除線。

此 skill 補充了 `2-jk-nb-consume`（自動投遞原始檔至 raw/）的工作——consume 處理 watcher 的入站流，而 inbox-digest 處理人工策展的升格清單。它通過封裝重複的周期性工作，降低人工操作錯誤（如意外刪除未升格項目）的風險，同時保持知識庫的秩序。

## 何時使用

- **定期排程**：每週自動執行，或搭配 `/schedule` 指令設定排程
- **手動觸發**：使用者說「跑 digest」、「整理 inbox」、「inbox 升格」
- **inbox 累積時**：條目數超過 10–15 項，需要集中升格處理
- **入庫清點**：定期清理過期的刪除線記錄

## 執行步驟

### 第 1 步：掃描並分類

讀取 jk-nb/inbox.md，識別三類條目：
1. **未標記項**（`- [ ] ...` 或 `- ...`）：待升格的候選條目
2. **已標記項**（`- [x] ...` 或 `~~...~~`）：已升格或放棄的條目
3. **過期標記**（刪除線且超過 30 天）：應該清除的紀錄

### 第 2 步：評估與升格映射

對每個未標記項，執行：
- **讀取內容**：提取條目的標題、描述、關鍵字
- **判斷有效性**：該條目是否仍有現實價值（非過時、重複或已實現）
- **映射目標位置**：根據內容類別（如 "React patterns" → `wiki/react.md`、"財報術語" → `wiki/finance.md`）決定目標
- **檢查目標頁存在性**：確認 wiki/ 下的目標檔案已存在；若缺失，記錄為「需手動建檔」

### 第 3 步：執行升格

對於有效條目且目標位置確認的項目：
1. 在目標 wiki/*.md 中定位合適的位置（按分類、字母順序或時間順序，遵循現有結構）
2. 複製或改述條目內容，與 wiki/*.md 的格式和風格一致（如列表、表格、段落）
3. 若原 inbox 條目包含來源連結或出處，在 wiki 版本中保留
4. 保存 wiki/*.md

### 第 4 步：標記已升格

在 inbox.md 中，對已升格的條目：
- 改為 `[x]` 或加上刪除線語法（`~~原條目~~`）
- 附註升格時間戳（可選，格式如 `~~原條目~~ (升格至 wiki/foo.md, 2026-06-13)`）
- 保存 inbox.md

### 第 5 步：提示清理

檢查已標記項中的過期記錄：
- 遍歷所有刪除線項，對比創建日期或標記日期
- 若標記日期 > 30 天，向使用者提示：「以下 X 項已超過 30 天，建議清除」
- 列出候選清除項（以刪除線格式或原始行編號顯示），讓使用者確認後手動清除

## 注意事項

**升格映射**：某些條目的歸類可能不明顯（跨多個領域、新興分類），此時應詢問使用者或標記為「需人工審視」，而非強制分類。

**目標檔案缺失**：若對應的 wiki/*.md 不存在，不應自動建檔。改為暫停升格，記錄「需新建 wiki/XXX.md」，由使用者確認。

**格式兼容性**：升格時應尊重目標 wiki 的現有格式（標題層級、列表風格、連結格式）；避免引入不一致的編寫風格。

**刪除線保留期**：30 天為預設保留期。若需調整（如 60 天），可在 skill 呼叫時傳入參數 `retention_days`。

**重複檢測**：升格前應檢查目標 wiki 頁面是否已包含相同或極相似的內容，避免重複條目。

**版本備份**：升格涉及文件修改，建議在執行前檢查 jk-nb 的 git 狀態；若有未提交的更改，先提交再執行 digest。

## 集成建議

搭配 `/schedule` 設定周期性執行（如每周一上午 9:00）：
```
/schedule jk-nb-inbox-digest --cron "0 9 * * MON" --args '{"retention_days": 30}'
```

或手動觸發：
```
/jk-nb-inbox-digest
```

集成 `/office-hours` 的「周清點」流程，在周回顧時一併執行 digest。
