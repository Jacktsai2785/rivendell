## Overview

**tej-sub-code-classifier** 是後端導向的輔助技能，用於確保公司產業分類任務在指派 TEJ（台灣經濟新報）官方子產業代碼時，遵循一致的**三段式輸入與結構化輸出**流程。

與內部自訂的 `taiwan-industry-classifier`（開放式分類）不同，本技能針對 **TEJ 官方子產業代碼系統**，需要精確對應 TEJ 代碼語義，並**強制使用 `StructuredOutput` 工具回傳結果**，避免模型直接輸出文字敘述。

### 為什麼需要這個技能

歷史會話分析顯示：
- Sessions 4、6、7 的三段式格式（公司代號 / 公司名稱 / 業務描述）在沒有 skill 引導下常遺漏最後的 stop hook 補呼叫
- Sessions 5、8、9 完全沒有 StructuredOutput 工具呼叫，模型直接回文字，不符合後端 agent 的結構化輸出需求
- 需要將此流程固化為技能，確保**每次都執行 → 評估 → StructuredOutput 回傳**


## 何時使用

當符合以下任一條件時，應觸發本技能：

1. **訊息明確包含 TEJ 代碼相關關鍵詞**
   - 例：「根據以下三段式資訊，請選出最合適的 tej_sub_code」
   - 例：「我想幫 2330 台積電分配一個 tej_sub_code，主要業務是晶圓代工」

2. **後端 agent 在進行台灣上市櫃公司分類時，需要正式的 TEJ 代碼指派**
   - 當輸入包含：公司代號（如 2330）、公司名稱、業務一句話描述
   - 當結果需要被下游系統解析（API、資料庫寫入、報表生成）

3. **避免直接文字回應的場景**
   - 任何涉及 TEJ 子產業代碼的分類任務，都應強制使用結構化輸出
   - 確保下游系統能可靠地解析分類結果


## 執行步驟

### 第一步：標準化三段式輸入

確保輸入包含完整的三元組資訊：

- **公司代號**：台灣上市櫃編號（例：2330、1101、6505）
- **公司名稱**：官方登記名稱（例：台積電、台泥、晟德）
- **主要業務描述**：一句話核心業務（例：「晶圓代工」、「水泥製造」、「生物製藥」）

若輸入不完整，向使用者明確請求補充資訊，提供範例格式：

> 請提供三段式資訊：
> - 公司代號：[如 2330]
> - 公司名稱：[如 台積電]
> - 主要業務：[如 晶圓代工]

### 第二步：查詢並理解 TEJ 子產業代碼

參考 TEJ 官方子產業代碼系統進行分類評估。TEJ 代碼採分層結構，常見範圍與示例：

- **2000–2999**：紡織纖維（2451 合纖、2454 化纖）
- **3100–3199**：電子零組件（3131 IC 設計、3132 晶圓代工、3140 封測）
- **3200–3299**：電腦與周邊（3231 監視器、3260 光磁儲存）
- **3300–3399**：半導體（3501 貿易、3504 光磊半導體製造）
- **4000–4999**：營造建材、金融保險等
- **9000–9999**：其他（貿易、服務等）

> **重要**：若執行時無法即時查詢最新完整的 TEJ 代碼表，應主動告知使用者並提供 TEJ 官方連結或建議人工確認。

### 第三步：評估並選定代碼

根據三段式輸入，判斷最符合的 TEJ 子產業代碼。評估因素：

- **主要業務性質**：製造、服務、貿易、金融等產業大類
- **次級產業分類**：細分領域（如晶圓代工 vs. IC 設計 vs. 封測）
- **歷史參考**：若公司已有既往 TEJ 代碼指派，優先保持一致（除非業務有重大變化）
- **多角化情況**：若涉及多個產業，選擇**主要業務**對應的代碼

### 第四步：使用 StructuredOutput 工具回傳

**必須**調用 `StructuredOutput` 工具，而非直接文字輸出。回傳結構如下：

```
{
  "company_code": "2330",
  "company_name": "台積電",
  "business_description": "晶圓代工",
  "tej_sub_code": "3132",
  "confidence": "high",
  "rationale": "台積電為全球領先晶圓代工公司，主要業務完全符合 TEJ 代碼 3132（晶圓代工）的定義。"
}
```

欄位說明：

- `company_code`：台灣上市櫃公司代號
- `company_name`：官方公司名稱
- `business_description`：使用者提供的業務描述
- `tej_sub_code`：選定的 TEJ 子產業代碼
- `confidence`：信心度標記（high / medium / low），指示分類的確定程度
- `rationale`：簡短說明選定此代碼的理由，包括主業特性或排除其他代碼的原因

### 第五步：驗證與持續改進

若下游系統、使用者或人工審核發現分類有誤，應：

1. 重新審視業務描述與 TEJ 代碼的對應關係
2. 考慮是否遺漏了公司的業務轉變或多元化
3. 查證是否有更精確的子代碼可適用
4. 記錄異議案例，持續改進分類邏輯


## 注意事項

### 已知限制

**TEJ 代碼表版本更新**
- TEJ 官方代碼表定期維護與調整，本技能基於既有知識進行分類
- 若新增公司或檢視到代碼變更，應手動驗證或外接最新 TEJ 代碼表
- 若發現本技能的分類與最新表規不符，應立即回報

**邊界與多角化情況**
- 多角化經營的大型企業（如鴻海、台塑）涉及多個產業，難以用單一代碼完全覆蓋
- 此時應選擇**主要獲利來源或營收占比最高的業務**對應的代碼
- 在 `rationale` 欄位明確說明：「主要業務為 X，次要業務為 Y（不含於此代碼）」
- 若需細化分類，應與使用者協商是否拆分為多個代碼或建立次代碼關聯

**非上市櫃公司**
- 本技能設計專用於台灣上市櫃公司（TWSE / TPEX）
- 若輸入為非上市櫃企業，應拒絕執行並說明：「本技能僅支持台灣上市櫃公司分類，該公司不在適用範圍內」

**StructuredOutput 的必要性**
- 歷史會話顯示，在沒有技能強制下，模型傾向直接回傳文字敘述
- 本技能的核心價值在於**確保每次都調用 StructuredOutput**，確保下游系統能可靠解析
- 不可省略此步驟

### 常見場景與應對

| 場景 | 應對方式 |
|------|--------|
| 輸入不完整（缺公司代號或業務描述） | 拒絕執行，列出需補充欄位，提供範例格式 |
| 公司主業不明確或多角化 | 選擇營收占比最高的產業代碼，在 `rationale` 說明多角化情況，信心度標記為 `medium` |
| 輸入非台灣上市櫃公司 | 拒絕執行，告知本技能的適用範圍限制 |
| 無法精確對應現有 TEJ 代碼 | 選擇最接近的代碼，標記信心度為 `low`，在 `rationale` 中詳述模糊之處 |
| 多個代碼都可能適用 | 優先選擇最具體與狹義的代碼，在 `rationale` 逐一說明為何排除其他選項 |
| 公司進行重大業務轉型 | 優先查詢最新財報確認主業，確保分類反映當前狀態而非歷史狀態 |

### 與相關技能的協作

- **`taiwan-industry-classifier`**：用於自訂產業分類（開放式），不強制 TEJ 代碼，適合非結構化場景
- **`tw-company-lookup`**：用於公司資訊查詢（代號、名稱、基本財務），建議先調用以驗證公司身份與當前營業狀況
- **後端公司情報流程**：本技能作為 backend agent 的標準化輔助，形成「查詢 → 驗證 → 分類 → 結構化輸出」的完整前處理流程
