# Skills Library — Project Rules

## Rivendell Operations

> Promoted 2026-05-13 from `.learnings/` — rivendell-meta rules used across the whole automation platform.

- **dashboard-next is launchd-managed**: Services `com.sk.dashboard.api`, `com.sk.dashboard.web`, `com.sk.dashboard.watchdog` are managed via `~/Library/LaunchAgents/com.sk.*.plist` with `KeepAlive=true`. To restart: `launchctl bootout gui/$(id -u) <plist>` → modify `.next/` if needed → `launchctl bootstrap gui/$(id -u) <plist>`. NEVER `kill` directly — launchd respawns mid-build → Turbopack runtime path corruption. Logs at `~/Library/Logs/sk-agent/com.sk.dashboard.*-stderr.log`.
- **Skills domain boundaries**: gstack-prefixed skills (`/gstack-*`) live in the gstack repo and own external-facing dev workflow. Rivendell skills own internal automation (agents, harvest, retro, sk-* tooling). Before creating a new skill, confirm which territory it belongs in — don't duplicate.
- **Built-in Claude Code skills are invisible to `bin/sk audit`**: Skills shipped with Claude Code itself (e.g., `/init`, `/review`, `/security-review`) don't appear in `~/.claude/skills/` and won't show in skill-audit reports. If a skill is "missing", check if it's a built-in before logging as a defect.
- **Storyline-first is a hard gate**: For B2B/presales deck work, `storyline.md` must reach `status: signed-off` via `/slide-office-hours` before any slide generation. This is intentionally NOT enforced by hook (would over-fire); enforce via prompt routing in `~/.claude/CLAUDE.md` Slide / Deck Building flow.
- **Never hardcode the repo/project name**: Derive `PROJECT_NAME` from `basename "$PWD"` or `git config remote.origin.url`. Plist labels (`com.sk.agent.<project>.<name>`), dashboard project lookup, and agents.conf all depend on this — hardcoded paths break on repo rename.
- **`reports/*` curation under auto-stage hook**: The auto-stage PostToolUse hook (`.claude/hooks/auto-stage-hook.sh`) stages every Edit/Write. `reports/` is generated by scheduled agents (harvest/test/skill-audit/etc.) — don't manually edit or commit them in interactive sessions; let the responsible agent own its files. Stale reports → archive to `reports/archive/YYYY-MM/`, don't delete.

## Post-change rule

After any skill change (add, remove, rename, modify description/trigger), update `README.md` to keep the Skills Catalog table in sync:
- Skill name, trigger method, and description must match the SKILL.md frontmatter
- Update the skill count in the "Skills Catalog" heading and the Structure tree
- If a new category is added, add it to both the Structure tree and a new table section

## Skill routing

When the user's request matches an available skill, ALWAYS invoke it using the Skill
tool as your FIRST action. Do NOT answer directly, do NOT use other tools first.
The skill has specialized workflows that produce better results than ad-hoc answers.

Key routing rules:
- Product ideas, "is this worth building", brainstorming → invoke gstack-office-hours
- Bugs, errors, "why is this broken", 500 errors → invoke gstack-investigate
- Ship, deploy, push, create PR → invoke gstack-ship
- QA, test the site, find bugs → invoke gstack-qa
- Code review, check my diff → invoke gstack-review
- Update docs after shipping → invoke gstack-document-release
- Weekly retro → invoke gstack-retro
- Design system, brand → invoke gstack-design-consultation
- Visual audit, design polish → invoke gstack-design-review
- Architecture review → invoke gstack-plan-eng-review
