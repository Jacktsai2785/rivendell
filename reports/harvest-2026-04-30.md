## Skill Candidates

### 1. `channel-partner-deck-pack` — **Moderate→Strong**

- **Purpose**：當提案要走通路商（中華電、資服公司、顧問公司、SI…）的品牌時，從通路 master deck 反向萃取一組「三件式 brand pack」：
  - `BRAND.md` — 視覺骨架、字級、色票、section divider、case study 頁規格（人類 / AI 都讀）
  - `tokens.json` — 程式可消費的 design tokens（給 html2pptx / template 套版）
  - `EXAMPLE-{n}-{vertical}.md` — 整本 deck 逐頁逐元素抄錄，含 storyline 主軸表 + 每頁元素座標 + 每個案例 customer 名稱（給 AI 後續 mirror 同產業時抓骨架）
- **Trigger when**：使用者說「把這份通路 deck 變成可重用的 brand」「中華電版/資服版的 deck 風格」「我要拿這份做食品業/紡織業 vertical 範例」「通路 master deck」「reverse extract a partner deck」「mirror this deck for {新客戶}」。
- **DO NOT TRIGGER**：要產通用 HTML template（→ `slide-template-extractor`，產 HTML，不產 markdown narrative）、從零做 deck（→ `pitch-deck` / `sales-material`）、單純讀 .pptx 內容（→ `office-pptx`）。
- **Category**：`docs/`（與 `slide-template-extractor` 同類）
- **Rationale**：Session 2（378 訊息 / 119 Edit / 22 Write）大量產出在 `cht-master/` 的 BRAND.md + tokens.json + EXAMPLE-2-FOOD-INDUSTRY.md 三件式骨架，且明確意圖是「之後光泉、其他食品業 prospect 都從這裡 pull」。既有 `slide-template-extractor` 產 HTML template（給程式套版），不涵蓋：
  - **整本逐頁的 markdown 抄錄**（給 AI 做敘事 mirror 用）
  - **vertical-specific examples**（同通路下不同產業要分別存一個 EXAMPLE 檔）
  - **通路 ↔ 終端客戶**的概念分層（master deck 屬通路、實際提案屬終端客戶）
- **與既有 skill 的關係**：`slide-template-extractor` 解決「樣式鎖定」，`channel-partner-deck-pack` 解決「敘事 + tokens + 多 vertical 範例」。可作為 sibling，或在 `slide-template-extractor` 加一個 `--mode=brand-pack` 模式。**建議獨立**——產出形態（markdown 而非 HTML）與下游消費者（AI agent 而非 html2pptx）足夠不同。
- **預期 ROI**：使用者主要 B2B 顧問業務跑 2-3 條通路（中華電、{未來其他通路}），每條通路平均要做 5-10 個終端客戶，brand pack 一次萃取後反覆 reuse；session 2 已花 378 訊息把第一次的方法論摸出來，下一次有 skill 就能 < 50 訊息完成。

### 2. `delivery-pdf-data-coverage` — **Moderate**

- **Purpose**：在 delivery 階段，當客戶丟一份新 `project_plan.pdf` / `requirements.pdf` 進來時，自動：
  1. 抽出 PDF 中所有提到的資料源 / 欄位 / 表名 / 代碼類別
  2. 對照當前 `DATA_TABLES.md` 與 `標準化代碼對照表*.md` 的覆蓋度
  3. 產出 `data_quality_report.txt`：「已覆蓋 / 缺欄位 / 標準化字典還沒收錄的 enum」三類缺口
  4. 把待確認問題歸入下次客戶會議的 `meetings/` agenda
- **Trigger when**：使用者說「這份 PDF 有提到的資料我們有嗎」「檢查資料來源覆蓋」「對照標準化代碼」「{某客戶}的 SR/系列/產品線資料齊不齊」，或新 PDF 落入 `02-Deliveries/*/3-delivery/` 子樹。
- **DO NOT TRIGGER**：PDF 是合約 / 報價（→ `sow-writer` / `rfq-writer`）、PDF 是廠商 datasheet 純讀取（→ `office-pdf`）、做 ETL pipeline（→ 不在範圍）。
- **Category**：`workflow/`
- **Rationale**：Session 3 雖只 32 訊息（Bash 23、Read 5）但工作模式很乾淨：PDF in → 比對既有 `DATA_TABLES.md` + `標準化代碼對照表_草案.md` → 產 `data_quality_report.txt`。立積電子的場景是封測資料（fm_code / rc_code），但抽象後對任何「客戶持續丟新需求 / delivery 已有資料骨架」的場景都通用（製造業 IoT、金融資料、業務分析…）。樣本只有一次但流程乾淨、產出物明確，故給 Moderate 而非 Strong。
- **與既有 skill 的關係**：`doc-to-structured-data` 是「把 .pdf/.doc/.xlsx 轉成 CSV/JSON」的通用解析；本 skill 是「PDF vs. 既有資料骨架的差異報告」——是 *檢查 / 對照* 而非 *轉換*。可建立在 doc-to-structured-data 的 PDF 解析能力之上。
- **建議**：先觀察是否第二次出現再建。若立積之後 KP / 高通 / 其他半導體客戶也走同樣 flow，就升 Strong；若沒有，留在 Peter 自己的 `02-Deliveries/CLAUDE.md` 當 local convention。
