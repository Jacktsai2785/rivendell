## 重大發現：大規模 Skill Routing Miss

本次最關鍵的觀察不是「有什麼新的 skill 候選」，而是**現有 skills 完全沒有被觸發**。

`-home-jacktsai-taiwan-company` 專案共 29 個 session，全部在做以下三件事：
1. `tw-company-website-finder`（11 個 session，sessions 5,6,8,10,18,20,22,25,26,28,29）
2. `tw-company-pe-memo`（16 個 session，sessions 2,3,4,7,9,11,12,13,15,16,19,23,24,27,30）
3. `taiwan-news-classifier`（1 個 session，session 21）

但只有 **session 14** 正確路由到 `tw-company-pe-memo`。其他 28 個 session 全部繞過 skill 直接執行。

**根本原因推測：** `taiwan-company` 是 headless agent 專案，intent 由外部程式組成 prompt 直接注入，沒有觸發 skill routing 的機制。Skills 設計為互動式 Claude Code 使用，不適用於 headless 批次模式。

## Skill 候選清單

### 1. `tw-company-batch-dd-orchestrator` — Moderate

**用途：** 對一批公司（列表）依序執行「查官網 → PE 盡調備忘錄」的完整流程，並將結果彙整到報告。

**觸發條件：** 使用者說「幫我對這批公司做盡調」、「批次 DD」、「把這份名單跑一遍 PE 分析」。

**分類：** `workflow/`

**理由：** Sessions 顯示目前執行模式是一次一家公司，分成「網站查找 session」與「DD session」兩種獨立批次。若有 orchestrator skill，可以收到公司列表後自動鏈接 `tw-company-website-finder` → `tw-company-pe-memo`，減少人工調度。

**現有 skill 覆蓋：** `tw-company-website-finder` + `tw-company-pe-memo` 各自存在，但沒有批次編排層。

---

### 2. `tw-company-name-extractor` — Weak

**用途：** 從一段原始文字（如新聞、文件、會議記錄）中提取所有公司名稱，回傳 JSON 陣列。

**觸發條件：** Session 17 的 intent 完全是這個操作。

**分類：** `backend/`

**理由：** Session 17 是一次性的 LLM 呼叫（無工具使用），可能是外部 pipeline 的一個子步驟。若需要在 Claude Code 內互動使用，不夠複雜到需要獨立 skill；但如果 `taiwan-company` 專案中有多個這類操作，值得作為 headless agent 的 prompt template 文件化。

**建議：** 先觀察這個 pattern 是否重複出現超過 5 次，再決定是否 skillify。

---

### 3. `headless-skill-bridge` — Weak

**用途：** 讓 headless agent 專案（如 `taiwan-company`）能夠正確觸發 rivendell skills，而不是繞過它們。

**觸發條件：** 建立新的 headless batch 專案時。

**分類：** `meta/`

**理由：** 本次最大的浪費是現有 skill 的知識無法被 headless 模式重用。若能設計一個「skill 呼叫橋接」協議（例如 headless intent 的 prefix 格式），未來 `tw-company-website-finder`、`tw-company-pe-memo` 等 skill 的邏輯就不用在 headless prompt 裡重複維護。

**注意：** 這更像是架構決策，而非 skill — 先確認 headless agent 的路由機制是否可改，再決定形式。
