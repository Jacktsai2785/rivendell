### Moderate — 有潛力但需更多樣本

#### 1. `presentation-analyzer` — 客戶簡報結構分析器

- **用途**: 接收客戶提供的簡報（PDF/PPTX），提取結構化摘要：主題、核心訊息、技術堆疊、預算暗示、組織架構線索、潛在合作切入點
- **觸發**: 「分析這份簡報」、「簡報拆解」、「看看客戶在說什麼」、使用者上傳客戶簡報檔案
- **分類**: `workflow/`
- **依據**: Session 1 花大量時間分析中華電信智慧製造簡報，提取技術重點和商業意涵。這與 `customer-intel`（從零研究公司）不同，是「已有客戶素材 → 提取情報」的反向流程。目前無 skill 覆蓋簡報分析。
- **待觀察**: 需確認此流程是否會在其他客戶案重複出現。若顧問業務持續成長，此模式應會高頻出現。

#### 2. `requirement-scenario-writer` — 需求情境文件撰寫

- **用途**: 根據客戶分析結果，產出結構化需求情境文件（含情境 A/B 比較、預期效益、技術可行性、風險評估）
- **觸發**: 「寫需求情境」、「情境 A/B」、「需求規劃」、在 customer-intel 或 presentation-analyzer 之後自動建議
- **分類**: `docs/`
- **依據**: Session 1 產出了 `2026-04-10-需求情境-A-B.md`，結構化地比較兩種解決方案情境。`requirement` skill 聚焦軟體需求（user story/acceptance criteria），而此需求情境更偏顧問業務的「提案前需求分析」。
- **待觀察**: 與 `sow-writer`（SOW 撰寫）和 `rfq-writer`（報價單）組成顧問業務文件鏈。需更多案例確認固定結構。

#### 3. `domain-knowledge-researcher` — 領域知識快速研究

- **用途**: 針對特定技術領域進行快速結構化研究，產出比較表（格式比較、技術選型、標準對照），輸出為 summary.md
- **觸發**: 「研究 X 和 Y 的差異」、「比較 A/B/C 三種方案」、「幫我搞懂 X 領域」
- **分類**: `workflow/`
- **依據**: Session 2 研究 ODB++ / Gerber / IPC-2581，使用 Agent + WebSearch 後產出結構化比較。這是顧問常見需求——快速建立某領域知識。
- **待觀察**: 與 `autoresearch` skill 有部分重疊（autoresearch 是 goal-directed loop），但 autoresearch 偏向「反覆驗證直到達標」，此處更像「一次性深度研究 → 結構化輸出」。需確認是否值得獨立成 skill 或作為 autoresearch 的子模式。

### Weak — 特徵不足或已有覆蓋

#### 4. `headless-agent-diagnostics` — Agent 排程健康診斷

- **用途**: 檢測 headless agent 啟動失敗模式，分析 1-msg session 的原因（超時、前置條件不滿足、plist 配置錯誤）

- **依據**: 6/12 sessions 只有 1 msg 就結束，全部是 headless 模式。但這更像是基礎設施問題而非可重複的工作流程。
- **建議**: 不建議成為 skill，但值得在 `headless-agent` 或 `launchd-agent` skill 中加入 diagnostic checklist section。
