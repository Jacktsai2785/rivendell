#### `large-file-refactor`

- **目的**：將超大單檔元件（500+ 行）系統化拆分為模組化架構，同時維持介面相容性
- **觸發條件**：使用者提到檔案超過 500 行、「monolith」、「模組化重構」，或要求拆分大型元件
- **分類**：quality
- **依據**：Session 16 專門處理 `deals/[id]/page.tsx`（1343 行）拆分。模式為：
  1. 讀取整個檔案，識別邊界（by 功能 / by route segment）
  2. 逐區提取子元件，保留 props 介面
  3. 建立 index 重新匯出，讓舊 import 路徑不變
  4. 驗證編譯與型別正確
  - 這個流程在不同技術棧（React、Vue）均適用，可高度複用
- **與現有 skills 差異**：`code-reviewer` 只做審查不重構；`simplify` 是清理邏輯，不是分拆大檔案

#### `vector-search-setup`

- **目的**：在 FastAPI 專案中從零建立向量搜尋知識庫，涵蓋 embedding 套件選型、資料建模、語意搜尋 API、索引管理
- **觸發條件**：使用者問「我的知識庫用什麼套件」、要做語意搜尋、想讓 agent 具備長期記憶、需要 RAG 架構
- **分類**：backend
- **依據**：Sessions 14–15（sales-assistant S40）明確出現「知識庫套件確認」→ embedding 實作→ `embedding.py` 與 `ai_provider.py` 撰寫。這是從問題診斷到完整實作的多步驟流程，需要特定知識（套件比較、chunk 策略、cosine similarity、索引更新）
- **與現有 skills 差異**：`db-migration` 做 schema 遷移；`knowledge-graph` 是 JSONL 實體記憶，不是向量搜尋

#### `multi-tenant-rbac`

- **目的**：在全端應用中實作多用戶角色權限系統，支援 Admin 全覽 / 成員自覽的內外帳模式
- **觸發條件**：使用者要設計「每個人看到的權限」、「管理員 vs 一般用戶」、「內外帳」、「role-based access」
- **分類**：backend
- **依據**：Sessions 1–2（Family-Fiscal）密集實作 admin-only.tsx、auth-context.tsx、auth-guard.tsx。模式為：JWT + role 欄位 → middleware guard → 前端 context → 條件渲染。這個模式在任何多用戶 app 都會重現
- **與現有 skills 差異**：`oauth-token-vault` 是 OAuth provider 整合；`security-review` 是審查不是建構；沒有現有 skill 專門教「從頭建 RBAC」
