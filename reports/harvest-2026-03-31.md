## Skill 候選分析

### Strong — 強烈建議建立

#### 1. `audio-transcription` — 音檔上傳與逐字稿轉換

- **用途**: 接受使用者上傳的音檔（mp3/wav/m4a），呼叫語音辨識 API（Whisper/AssemblyAI/Deepgram），回傳結構化逐字稿並存檔
- **觸發**: 使用者說「我要上傳音檔轉逐字稿」、「audio transcription」、或前端出現音檔 input 且後端無轉錄邏輯
- **建議分類**: `backend`
- **理由**: Session 10 完整走過此流程，且為高重用場景：
  1. 前端：`<input type="file" accept="audio/*">` + 進度顯示
  2. 後端：`multipart/form-data` 接收 → 暫存 → 呼叫 Whisper API
  3. 結果存入 DB（`transcript_text`, `language`, `duration`）
  4. 串流回傳逐字稿或輪詢機制
  5. 錯誤處理（檔案格式、大小限制、API 逾時）
- **無現有 skill 覆蓋**: `doc-to-structured-data` 處理文件，不處理音檔；`imap-smtp-integration` 不相關

---

#### 2. `ai-data-layer-split` — 後端 AI/Data 分層重構

- **用途**: 將 FastAPI 後端中「純資料管理 CRUD」與「AI/Agentic 邏輯」明確分成兩層，避免混雜造成耦合與可測試性差
- **觸發**: 使用者說「把 AI 跟資料管理分開」、「重構 agent.py」、或單一檔案同時包含 DB query 和 LLM call
- **建議分類**: `backend`
- **理由**: Sessions 7 & 10 重複出現此重構需求，模式固定：
  1. 盤點哪些函式是純 CRUD（`get_deal`, `update_client`）
  2. 哪些是 AI 邏輯（`generate_embedding`, `call_llm`, `run_agent`）
  3. 建立 `data/` 層（repository pattern）與 `ai/` 層（provider pattern）
  4. `ai_provider.py` 統一管理 LLM 呼叫（模型選擇、retry、cost logging）
  5. 驗證 `embedding.py`、`deals.py`、`agent.py` 之間的依賴方向正確
- **與現有 `large-file-refactor` 差異**: `large-file-refactor` 處理單一大檔案切分，本 skill 聚焦架構層次的職責分離語義

### Moderate — 值得考慮

#### 3. `multi-user-rbac` — 多成員角色權限系統

- **用途**: 在家庭/小組應用中實作角色型存取控制（RBAC），含內外帳可見性、成員管理、auth-context + auth-guard 前端模式
- **觸發**: 使用者說「加入成員管理」、「內外帳權限」、「不同人看不同資料」、或 app 有多個使用者但共用帳號
- **建議分類**: `backend`（含前端 guard 模式）
- **理由**: Session 2（Family-Fiscal）花費超過 325 則訊息實作此系統，涵蓋：
  - `auth-context.tsx`（React Context 管理登入態與角色）
  - `auth-guard.tsx`（路由層 HOC，依角色封鎖頁面）
  - `admin-only.tsx`（元件層條件渲染）
  - 後端 API 以 JWT role claim 決定回傳欄位（外部帳隱藏內部欄位）
  - 家庭成員 CRUD（新增/移除成員、角色變更）
- **保留疑慮**: 此模式與 Supabase RLS / Firebase Security Rules 細節有差異，若加入平台分支會更有價值；再觀察 1–2 個 session

---

#### 4. `cross-device-deploy` — 跨設備一鍵部署

- **用途**: 以一個中央 repo（如 rivendell）為核心，在任何設備 `git pull` 後透過單一腳本啟動所有子專案（含環境設定、依賴安裝、服務啟動）
- **觸發**: 使用者說「在另一台設備部署」、「一鍵啟動所有服務」、「pull 下來後怎麼跑」
- **建議分類**: `workflow`
- **理由**: Session 9 完整實作此流程：
  1. `bootstrap.sh`：讀 `.env.example` → 提示填入機密 → 建立 `.env`
  2. `start-all.sh`：迭代子目錄 → `npm install` / `pip install -r requirements.txt` → 後台啟動服務
  3. `health-check.sh`：curl 各服務 port → 確認存活
  4. launchd plist 或 systemd service 設為開機自啟
- **與現有 `deploy` 差異**: `deploy` 聚焦平台部署（Fly.io/Vercel），本 skill 聚焦「本機多專案一鍵啟動」場景

---

#### 5. `opportunity-scanner` — 機會自動發掘 Agent

- **用途**: 自動搜尋網路上的商業/投資/副業機會，分析是否可由 Claude 自動執行，整理優先清單並設定自動執行流程
- **觸發**: 使用者說「找賺錢機會」、「分析哪些可以自動化執行」、「持續掃描機會」
- **建議分類**: `workflow`
- **理由**: Session 4（curia）是完整案例：
  1. WebSearch 蒐集機會來源（副業、外包、套利機會）
  2. 以結構化 rubric 評分（可自動化程度、風險、所需資源）
  3. 存入排優先順序的 `opportunities.json`
  4. 高分機會觸發 sub-agent 執行
  5. 每日定期執行（headless-agent + launchd）
- **保留疑慮**: 部分步驟已由 `investment-research` + `headless-agent` 組合覆蓋；建議等 curia 再成熟 1–2 個 session 後再決定
