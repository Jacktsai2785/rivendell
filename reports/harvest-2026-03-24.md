1. `lsof -ti:<port>` 找出佔用 port 的 process

## Skill 候選評估

### Moderate — 服務編排器 (service-orchestrator)

- **用途**：管理多專案的本地開發服務生命週期（啟動、停止、重啟、健檢）
- **觸發**：`/restart`、「重啟服務」、「啟動 backend」、開發中需要重啟服務時
- **分類**：`workflow/`
- **理由**：
  - 4+ 個 session 重複執行完全相同的 kill → start → verify 流程
  - 每個專案的 port 配置不同（rivendell: 3000/8000, sales-assistant: 3002/8002）
  - 目前每次都手動拼 bash 指令，容易遺漏步驟（如忘記啟動 telegram bot）
- **風險**：
  - 各專案啟動方式差異大，泛化困難
  - 部分專案已有 start scripts，skill 可能只是薄薄的包裝
  - 可考慮改用 project-level CLAUDE.md 中記錄啟動指令即可
- **結論**：Moderate。如果專案數量繼續增加或啟動步驟更複雜，值得做成 skill。目前可先在各專案的 `.claude/commands/restart.md` 統一格式。

### Weak — Skills/Hooks 健康檢查 (agent-health-check)

- **用途**：檢查 Claude Code 環境的 symlinks、hooks、settings 是否完整
- **觸發**：「檢查 hooks」、「skills 是不是斷了」、環境遷移後
- **分類**：`meta/`
- **理由**：
  - 2 個 session 花費大量時間診斷斷裂的 symlinks 和 hooks
  - 路徑遷移（skills-test → rivendell）是已知的 breaking change 來源
- **風險**：
  - 已有 `audit-fix` skill 處理 permissions 審計
  - 環境遷移是低頻事件，做成 skill 的 ROI 低
  - `sk-agent-doctor` launchd agent 已在處理部分健康檢查
- **結論**：Weak。建議將 symlink 修復邏輯整合到現有的 `audit-fix` skill 或 `sk-agent-doctor` 中，而非新建 skill。

### Weak — 任務狀態監控 UI (task-monitor)

- **用途**：為系統管理頁面加入即時任務狀態顯示
- **觸發**：Session #8 中使用者要求「執行後看到目前執行項目」
- **分類**：不適合做成 skill
- **理由**：
  - 僅出現一次，且高度耦合 news-stock 的 TaskManager API
  - 屬於專案特定的 UI 功能，不具跨專案可重用性
- **結論**：Weak。不建議做成 skill。
