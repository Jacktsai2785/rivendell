## Skill 候選清單

### 🟢 Strong

---

#### 1. `rbac-auth` — 角色權限存取控制

**目的**：在 React/Next.js + FastAPI 專案中實作 RBAC（Role-Based Access Control），包含 admin/user 分層視圖、JWT 驗證、ProtectedRoute 元件、後端 permission guard。

**觸發時機**：
- 使用者說「加入權限管理」、「admin 看不同畫面」、「內外帳分開」
- 新專案需要 login + 多角色視圖
- 存在 auth-context.tsx / auth-guard.tsx 但缺乏 RBAC 邏輯

**分類**：`backend`

**理由**：Sessions 1 & 2 完整展示了一個可複用的 RBAC 建置流程：
- 後端：FastAPI JWT + role 欄位 + middleware guard
- 前端：AuthContext → AuthGuard → 條件渲染（admin-only.tsx 模式）
- 出現相同模式的檔案：`auth-context.tsx`, `auth-guard.tsx`, `admin-only.tsx`, `api.ts`
- 此模式在幾乎所有多用戶 app 中都需要，現有 skills 中無對應（`oauth-token-vault` 僅涵蓋 OAuth token 儲存，`security-review` 是 checklist 而非實作模板）。

---

#### 2. `multi-project-deploy` — 多專案一鍵部署

**目的**：以單一 repo（如 rivendell）為核心，在新機器上 pull + 啟動多個獨立專案，支援 Docker 編排、環境變數注入、部署狀態 dashboard。

**觸發時機**：
- 使用者說「在新電腦部署所有專案」、「一鍵部署」、「pull 下來後自動啟動」
- 有多個 sibling repos 需要協調啟動順序和環境
- 需要管理多個 `.env` 來源

**分類**：`workflow`

**理由**：Session 13（362 則訊息）是本次 harvest 最長的 session，完整展示了一個跨專案部署協調系統的設計與實作：
- Dockerfile 多專案編排
- 環境變數統一注入機制
- 部署狀態 dashboard（AgentCard.tsx、CollaborationFlow.tsx）
- 現有 `deploy` skill 針對單一專案，`tunnel-proxy-deploy` 針對 Cloudflare Tunnel；此 skill 填補「多專案協調」的空缺。

## 重複模式觀察

1. **FastAPI + Next.js 全棧開發**：Sessions 1/2/13 都在同一個技術棧上反覆建立類似的結構（.env, .gitignore, 前後端分離 port），但缺乏 RBAC 這塊的標準化 → 支持 `rbac-auth` 優先度。

2. **Headless Agent 執行模式**：Sessions 4/8/9 都是「啟動 skill → 執行 → 輸出 report」的無人值守模式，且都依賴 `headless-agent` + `launchd-agent`。這個組合已有良好支援，無需新增 skill。

3. **知識庫 md-centric 設計**：Sessions 10/11 顯示 `markdown-file-ssot` 是銷售助手的核心設計原則，且被反覆應用。現有 skill 已存在，使用頻率符合預期。
