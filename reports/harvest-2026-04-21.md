## Skill Candidates

### Strong

#### 1. `env-doctor` — 環境健檢腳本生成器
- **Purpose**：為專案產生 `doctor.sh`（或 `doctor.py`），檢查 Python/Node 版本、依賴鎖定檔 hash、模型/資料下載狀態、key env vars、外部服務連線；輸出彩色 PASS/FAIL 報告供跨機器一鍵診斷。
- **Trigger**：使用者說「為什麼我在另一台機器跑出來不一樣」「環境排錯」「reproducibility」「寫一個 doctor 腳本」「跨機器 setup 驗證」。
- **Category**：`backend/` 或 `quality/`
- **Rationale**：Session 1 花了大量時間建立 `doctor.sh`，這是任何多人/多機器專案的通用痛點。現有 `qa-testing`、`qa-auto` 處理的是測試，不是環境一致性；`init-project` 只負責初始化，不做 runtime health。值得獨立 skill。
- **是否存在**：否（已檢查 `skills/` 全部子目錄）。

#### 2. `repro-exam` — 跨機器再現性測驗考題產生器
- **Purpose**：依照專案的核心邏輯（如 backtest engine、portfolio strategy）產生一組 deterministic 測驗（input → expected output），讓對方在自己的機器跑完比對結果，快速定位差異來源（資料源 / 套件版本 / 隨機種子 / floating point）。
- **Trigger**：使用者說「我有沒有標準的測驗考題」「跑出來不一樣怎麼排錯」「驗證對方的計算結果」「reference output for comparison」。
- **Category**：`quality/`
- **Rationale**：Session 1 中使用者明確要求「標準的測驗考題可以讓對方的電腦排查」，這是與 `qa-testing` 不同的問題：不是驗證程式正確性，而是驗證「兩台機器是否產生相同結果」。常見於量化研究、ML reproducibility、科學運算。
- **是否存在**：否。

### Moderate

#### 3. `local-port-registry` — 本機多專案 port 管理
- **Purpose**：維護 `~/.claude/port-registry.md`（或專案 `.ports`）記錄每個專案的固定 port（news=3001、rivendell=3000、…），啟動前檢查衝突，自動修正 `next.config.ts` / `package.json` / `vite.config.ts` 的 port 設定。
- **Trigger**：使用者說「打開怎麼變成 X」「port 衝突」「之前有規定的 port」、或開新前端專案時。
- **Category**：`workflow/`
- **Rationale**：Session 3 出現典型「打開 localhost:3000 怎麼跳到 news」場景，是同時跑多個 Next.js / FastAPI 專案的人都會遇到的問題。目前無對應 skill。Moderate 因為本身範圍較窄，但場景相當高頻。
- **是否存在**：否。
