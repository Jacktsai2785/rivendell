## 已觀察的工作流程模式

### 模式 A：公司名稱擷取（6 次）
**輸入：** 含公司名稱的任意文字（新聞、名冊、報告）
**輸出：** JSON 陣列 `["公司A", "公司B", ...]`
**規則固定：** 含「股份有限公司/有限公司」者完整列出；疑似機構名但無標準結尾的也列出；禁止捏造

### 模式 B：官方網站搜尋（10 次）
**輸入：** 公司名稱 + 統一編號
**輸出：** 單一 URL 字串（找不到則空字串）
**搜尋策略：** `WebSearch "公司名稱" → 評估結果 → 回傳最可能官網`
**注意：** 暢橘文化（#7）需 4 次 WebSearch 才確認；社會網絡（#8）因名稱太通用需用統一編號輔助

### 模式 C：PE 投資備忘錄（14 次）
**輸入：** 公司基本資料（名稱、統編、設立日期、資本額、所在地等）
**研究步驟：**
1. WebSearch 公司名稱 → 官網/新聞/LinkedIn
2. WebFetch 官網（About 頁、團隊頁、產品頁）
3. WebSearch 「公司名稱 + 新聞/募資/創辦人」
4. WebFetch MOPS / Crunchbase / 媒體報導
5. 綜合輸出結構化 DD 備忘錄（市場、競爭、財務、風險、投資觀點）

## Skill 候選清單

### ✅ `tw-company-name-extractor` — **Strong**

**目的：** 從任意非結構化文字中擷取台灣公司/機構名稱，回傳 JSON 陣列
**觸發條件：** 給定文字、需要識別其中的公司名稱、輸出 JSON 格式
**分類：** `backend`（utility）
**理由：**
- 在 30 個 session 中出現 6 次，均為一模一樣的 prompt 結構與輸出格式
- 無工具依賴，純 LLM 推理，速度快、成本低
- 是批次 DD pipeline 的第一步，也可獨立用於新聞解析、合約分析等情境
- **現有 skills 中無此能力**（`tw-company-lookup`、`tw-company-identify` 都需要已知公司名稱）

**輸出規格：**
```json
["公司名稱A", "公司名稱B"]   // 找不到時回傳 []
```

### `tw-company-batch-dd-pipeline` — **Moderate**

**目的：** 批次對多家台灣公司執行完整 DD 流程：文字擷取名稱 → 搜尋網站 → 逐一產出 PE 備忘錄
**觸發條件：** 使用者提供一份名單（PDF、文字、試算表），需批次產出多份 DD 備忘錄
**分類：** `workflow`
**理由：**
- 這 30 個 session 的整體模式正是此流水線在運行
- 三個步驟（name-extractor + website-finder + pe-memo）各自已有/將有 skill，缺的是協調層
- 若以 headless agent 實作，可並行處理多家公司（每家一個 sub-agent）
**風險：** 每次 session 成本約為 WebSearch × 7 + WebFetch × 5；100 家公司需預估成本後再啟動
