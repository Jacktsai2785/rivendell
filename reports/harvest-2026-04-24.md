## Skill Candidates

### 1. `solution-catalog` — **Strong**

- **Purpose**：以 frontmatter 驅動的產品 offering 目錄骨架，支援 `platform → offering → addon` 三層分類，並在每個 solution 頁面自動維護「被哪些客戶引用」的反向索引。
- **Trigger when**：使用者說「建立 solution 目錄」「產品規劃目錄」「offering 分類」「解決方案知識庫」，或正在建立多個 `materials/solutions/*.md` 檔案且需要反向引用。
- **DO NOT TRIGGER**：單純要組裝某客戶的提案（→ `sales-material`）、研究某家公司（→ `customer-intel`）、做 markdown SSOT schema 設計（→ `markdown-file-ssot` 先行）。
- **Category**：`workflow/`
- **Rationale**：Session 3 顯示使用者在 115 則訊息中用 47 次 Write 建立大量解決方案檔案（analog-gauge-ocr、chiller-energy-optimization、dashboard…），且明確要求「反向索引」與三層分類法（platform/offering/addon）。既有 skills 有缺口：
  - `sales-material` — **消費**素材庫來組裝簡報，不負責目錄**建立**
  - `markdown-file-ssot` — 提供通用 schema 原則，但不涵蓋 offering 專有的 platform/offering/addon 分類與反向索引模板
  - `presales-pipeline` — 管理客戶 pipeline，不管 solution 側
- **潛在自動化**：可附一支 `rebuild-backrefs.sh`，grep `solutions/` 下所有檔案的 `clients:` frontmatter 欄位，回寫每個 client 檔案的 `related_solutions:`。

### 2. `channel-partner-intake` — **Moderate**

- **Purpose**：當 prospect 是透過通路商（中華電、資服公司、顧問公司等）媒介接觸時，建立「`YYYYMM_通路商/終端客戶名/`」雙層資料夾結構，並以日期為前綴生成首次通話紀錄 + 初始需求文件。
- **Trigger when**：使用者說「新 client 透過 X 通路」「這是 X 介紹的案子」「202604_中華電」這類月份_通路前綴，或 first-call 筆記命名含通路資訊。
- **DO NOT TRIGGER**：直接接觸的一般客戶（→ `presales-pipeline` 原本的 `01_presales/<client-slug>/`）。
- **Category**：`workflow/`
- **Rationale**：Session 2 中使用者明確建立 `202604_中華電/鼎漢` 結構並表示「你能理解這個邏輯嗎」，表示這是刻意的歸檔方式。既有 `presales-pipeline` 採平坦 client-slug 結構，不表達「通路商」這個中介節點。這可以是 `presales-pipeline` 的擴充分支，也可以獨立為 channel 專屬 skill。
- **替代方案**：不新增 skill，而是在 `presales-pipeline` 中加一個「通路媒介」模式。但該 skill 已有明確 `01_presales/` 結構約定，混入會破壞既有語意。建議獨立。

### 3. `demo-deck-sync` — **Weak→Moderate**

- **Purpose**：客戶 demo / 電話結束後，把對話中浮現的新定位、新優勢（如「豁免未實作的自由度」）同步寫回 `demo-script.md`、`STATE.md`，並觸發 `build` 確認 dashboard 與簡報一致。
- **Trigger when**：使用者剛完成 demo / 客戶通話後說「要調整簡報」「加到簡報裡說明」「demo 後更新」。
- **Category**：`docs/`

- **建議**：不建立 skill；改寫到 ChimesAI 專案自己的 `AGENTS.md` 作為 local convention。

### 4. `client-call-notes` — **Weak**
