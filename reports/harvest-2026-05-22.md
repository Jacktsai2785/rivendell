## Skill 候選清單

### 1. `tej-sub-code-classifier` — **Strong**

**目的**：給定台灣上市櫃公司代號、公司名稱與一句話業務描述，從 TEJ 子產業代碼清單中選出最合適的 `tej_sub_code`，並透過 `StructuredOutput` 工具回傳。

**觸發條件**：
- 訊息包含「tej_sub_code」或「請選出最合適的 tej_sub_code」
- working directory 為 `web-mops-rev-backend`
- 任何公司分類到 TEJ 子類別的需求

**類別**：`backend/`（或 `workflow/`）

**理由**：
- Sessions 4、6、7 顯示**相同的三段式輸入格式**（公司代號 / 公司名稱 / 主要業務）和相同的輸出動作（StructuredOutput）
- Session 7 中 stop hook 強制補呼叫，顯示這個步驟常被遺漏 → 需要固化為流程
- Sessions 5、8、9 完全沒有工具呼叫，說明沒有 skill 引導時模型會直接回文字，不符合 backend agent 的結構化輸出需求
- 與現有 `taiwan-industry-classifier`（自動歸類到自訂產業清單）不同：此技能針對 **TEJ 官方子產業代碼**，需要知道 TEJ 代碼語意，且強制走 StructuredOutput
- 不重複任何現有 skill

### 2. `jk-nb-inbox-digest` — **Moderate**

**目的**：處理 jk-nb vault 的 `inbox.md`，對每個「仍活著」的條目 promote 到對應的 `wiki/*.md`，並在 inbox.md 用刪除線標記（不刪除原文）；提示使用者刪除超過 30 天的已劃線條目。

**觸發條件**：
- 使用者說「跑 digest」「整理 inbox」「處理 inbox.md」
- working directory 是 `jk-nb` 且提到 inbox

**類別**：`workflow/`

**理由**：
- Session 1 顯示這個工作流程已有明確規則（promote、刪除線、30 天提示），但未成文為 skill
- 與現有 `2-jk-nb-consume` **不同**：consume 處理 `raw/` 目錄（frontmatter 路由到 wiki），inbox-digest 處理 `inbox.md`（單一檔案，不同語意），二者是 jk-nb vault 的兩個獨立動詞
- 規則非顯然（保留原文 + 刪除線，而非刪除）且有時間門檻邏輯（30 天），適合固化

### 3. `mops-business-description-scraper` — **Moderate**

**目的**：從 MOPS 網站 `t05st03` 頁面爬取台灣上市/櫃/興櫃公司的「主要業務」（主營業務描述），支援單公司查詢與批次查詢，結果存入本機資料庫或回傳 JSON。

**觸發條件**：
- 使用者說「查公司業務描述」「從 MOPS 抓主營業務」「t05st03」
- 任何需要公司業務描述但 mops_databases 中找不到的情況

**類別**：`backend/`

**理由**：
- Session 11 中大量 Bash 操作嘗試對 `https://mopsov.twse.com.tw/mops/web/t05st03` 進行爬蟲，花費相當時間在解析 HTML 格式、處理 encoding、batch 策略
- 與現有 `mops-rev-scraper`（月營收）、`mops-financial-scraper`（財務報表）明顯不同：此為「公司基本資料」類別，尤其是業務描述欄位
- `tw-company-lookup` 提供查詢介面但不包含爬蟲邏輯
- 爬蟲細節（URL pattern、表格 XPath、encoding 坑）值得固化，避免每次重新摸索
