## Skill 候選評估

### 1. `client-email-to-artifacts` — **Strong** ✅

**用途**: 接到客戶 email（含附件或技術文件）時，一次展開產出三類 artifacts：
1. `YYYY-MM-DD-docs-received.md` — 文件摘要與 key takeaways
2. `YYYY-MM-DD-client-update-email-draft.md` — 回客戶的進度更新草稿
3. `YYYY-MM-DD-internal-meeting-brief.md` + `YYYY-MM-DD-team-handoff.md` — 內部會議 brief 與團隊交棒

**觸發**: "客戶寄來...", "收到 API 文件", "幫我做進度更新 email", "team handoff", "client update draft", "把這封信拆成內外兩份"
**分類**: workflow
**理由**:
- Session 1 用 67 msgs（全 session 最多）做這件事，且產出 4 份明確的 artifacts — 高工作量、高結構化
- ChimesAI presales 每週有 2-3 個類似 email thread（Sabre、台塑、力成都會發生），高頻重用
- 現有 `customer-intel`（公司背景研究）、`discovery-interview`（需求訪談）、`sow-writer`（合約文件）都不涵蓋「email thread in-the-middle」階段
- 可搭配現有的 `office-docx`/`gdoc-report-builder` 做格式化
- **結論**: 建議建立。SKILL.md 骨架就緒時可以先把 Session 1 的四份 artifact 當範本存進 reference

### 2. `long-running-task-monitor` — **Moderate** ⚠️

**用途**: 啟動一個長跑任務（backfill / deploy / CI / 資料分析）後，用 `ScheduleWakeup` 動態間隔回檢進度、記錄學習、完成時彙報。
**觸發**: "爬到今天", "監控 backfill", "watch this long job", "check progress in N minutes"
**分類**: workflow
**理由**:
- Session 3 用 181 msgs + 4 次 ScheduleWakeup 做這件事，顯示此 pattern 的長度與複雜度
- `headless-agent` 聚焦排程執行（launchd/cron 層），`autoresearch` 聚焦 verify-loop，`loop` skill 聚焦固定間隔 — 都不直接覆蓋「啟動 → 動態回檢 → 結束」
- **但**：ScheduleWakeup 本身的使用模式已經被系統 prompt 大幅規範（快取 TTL、delaySeconds 選擇），額外 skill 的增量價值待驗證
- **結論**: 觀望 — 若下次 harvest（04-22 / 04-24）再觀察到類似 181-msg ScheduleWakeup 監控流程，立即 codify。先把 Session 3 的 LEARNINGS 當作 reference 保存
