- Session 13 (Projects root) 再次出現相同需求，且有獨立規劃文件：`audio-transcription-flow.md`、`audio-transcription.html`、`audio-transcription.md`
- 兩次都涉及相同流程：前端上傳 UI → 後端音訊處理 endpoint（Whisper/speech-to-text）→ 顯示逐字稿結果

3. 儲存逐字稿到 DB，回傳結構化結果
4. 前端顯示逐字稿，支援編輯修正

#### 2. `rbac-permissions`

| 欄位 | 內容 |
|------|------|
| **名稱** | rbac-permissions |
| **分類** | backend |
| **目的** | 設計並實作角色型存取控制（RBAC），含前端 AuthGuard 與後端 API 驗證 |
| **觸發條件** | 「權限管理」、「RBAC」、「內外帳」、「role-based」、「每個人只能看自己的」 |
| **評級** | Strong |

**依據**：
- Session 3 (Family-Fiscal) 完整實作 admin/member 兩層權限：內帳（全家可見）vs 外帳（個人帳戶）
- 產出了 `admin-only.tsx`、`auth-context.tsx`、`auth-guard.tsx`、`api.ts` 等典型 RBAC 元件組合
- 此模式（AuthContext + AuthGuard + role-based route protection + API middleware）在任何多人系統都需要
- 目前無 skill 涵蓋此模式（`firebase-backend` 較偏向 Firebase 特定架構）

**核心步驟模式**：
1. 定義 role 枚舉（admin / member / viewer）
2. AuthContext：登入狀態 + role 存入 context
3. AuthGuard component：route-level 保護
4. API 層：middleware 驗證 token + role
5. UI 層：`admin-only.tsx` 等 wrapper component 控制顯示
