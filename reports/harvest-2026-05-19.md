## Skill 候選

### Strong

#### 1. `demo-data-anonymizer`
- **Purpose**: 將真實客戶資料（含敏感欄位）去識別化，輸出成 demo-ready 的 wide table，可直接灌進 Tukey/BI 等展示平台。包含：欄位敏感度判讀、語義保留決策（如颱風名稱該不該留）、長格式 → 寬格式 reshape、樣本驗證。
- **Trigger**: 「demo 用的資料」「去識別化」「假資料 demo」「anonymize 真實 data」「客戶資料拿來示範但要遮罩」
- **Category**: workflow
- **Rationale**: Session 2（189 訊息、83 Bash、33 Write、17 Edit）是這次 harvest 最重的單一工作流，且這個情境是 B2B 銷售/POC 反覆出現的：拿一個真實客戶的歷史資料做 demo，需在「保留洞察價值」與「不洩露身分」之間取捨。現有 `customer-intel` 是對外蒐集、`metadata-workshop` 是 schema 設計，沒有專門處理 demo 用的去識別化工作流。可以條列出常見決策 checklist（人名/廠別/日期偏移/聚合層級/外部因子保留），降低每次重新踩坑的成本。

#### 2. `executive-architecture-diagram`
- **Purpose**: 把工程版系統架構圖（含 port、品牌、英文縮寫）翻譯成「給長官/委員/CIO 看」的版本：中文化、隱藏品牌、內外網/DMZ 用獨立色塊分區、官方文件章節代碼當頂層 framing、headless Chrome 自截圖驗證。
- **Trigger**: 「長官版架構圖」「給委員看的圖」「不要太多英文 / 看不懂 443」「客戶老闆要看」「審查委員版本」「政府計畫書架構圖」
- **Category**: docs（或當作 `chart-design` 的 sub-workflow）
- **Rationale**: Session 3（114 訊息、38 Bash、25 Edit）整段在處理同一張圖反覆 v2 → v3 → cache 的痛苦過程，且根目錄 CLAUDE.md「Diagram & Slide Output Defaults」章節已經有 7+ 條規則是針對這個情境（16:9 1600×900、字級基準、官方文件 framing 優先、ABCD overlay 不主軸切版、headless 自截圖檢查、長官版 vs 工程師版分流）。規則散在 CLAUDE.md 裡，每次仍要靠模型記得 routing；包成 skill 後可以提供：(a) 雙版本 layout template（工程師版/主管版併存於 cache/），(b) WBS overlay 樣式庫，(c) headless 驗證 checklist。`chart-design` 是泛用圖表，這個專注於「給高階主管/外部審查」的架構圖場景。
