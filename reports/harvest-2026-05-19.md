## Skill 候選評估

### 🟢 Strong：`portable-project-snapshot`（可攜式專案快照）

- **Purpose**：把一個本地開發中的全端專案（程式碼 + DB + materials + .env）打包成一份可在另一台機器（同事的 Mac、客戶 WSL、新開發機）解壓即跑的 tarball，附自動產出的 `RESTORE-README.md` 寫明每一步還原指令、環境差異 checklist、常見踩坑。
- **Trigger**：
  - 「我要把資料轉移到另一台機器/WSL/同事」
  - 「打包輸出到 Downloads」
  - 「export 給別人用」
  - 「frozen snapshot」、「portable copy」
  - 「準備給客戶試跑」
- **Category**：`workflow/`（與 `deploy/` 區隔：deploy 是給 cloud/server；這個是給人，靠 USB / OneDrive / SCP 拷貝）
- **Rationale**：
  - **n=2 occurrence**：Session 2 主動執行（5/19 早上）；Session 3 接續處理「在新機器跑時欄位儲存失敗」（5/19 下午）— 完整地走過「打包→搬運→回報差異→補修」整個 lifecycle。
  - 既有 skill 不覆蓋：
    - `/deploy` 假設目標是 cloud platform (Fly.io / Vercel / Docker / Railway)，產出是 Dockerfile + CD workflow，不是離線 tarball。
    - `/tunnel-proxy-deploy` 是把本機服務暴露給外網，不需要搬運。
    - `/db-migration` 只處理 schema/data，不含程式碼+materials+env。
    - `/sqlite-to-postgres` 是 engine 換代，不是檔案搬運。
  - 共通 deliverable 已成形：`RESTORE-README.md`（Session 2 實際產出檔名）→ 可作為 skill template。
  - 共通踩坑點（Session 3 揭露的）：欄位儲存失敗 = 環境差異（Python venv / pip 版本 / node_modules / SQLite WAL 檔案沒帶到 / .env.local 沒打包），值得在 skill 的 README template 內預先列 checklist。
- **Skill 結構建議**：
  - `SKILL.md`：trigger 條件 + 兩階段流程（freeze: 打包；thaw: 還原）
  - `templates/RESTORE-README.template.md`：依專案型態（純 Python / Node 全端 / SQLite vs Postgres / 含 materials）分支
  - `scripts/snapshot.sh`：產出 tarball、附 manifest（含 sha256、檔案大小、產出時間、來源機器資訊）
  - `checklists/cross-env-diffs.md`：Mac↔WSL、Python venv 版本、SQLite WAL、絕對路徑 hardcode 等檢查表
- **與既有 skill 的關係**：
  - 可作為 `/dispatching-parallel-agents` 之前的「環境就緒」前置 skill
  - 拆解大檔案時 (`/large-file-refactor`) 若涉及搬遷，可先做 snapshot
  - 跨機器交接後若仍有 bug，rollback 到 snapshot
